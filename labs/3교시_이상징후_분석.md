# 실습 3: 이상 징후 분석

## 목표

KQL(Kibana Query Language)을 활용하여 수집된 로그에서 공격 패턴을 검색하고, 정상 트래픽과 공격 트래픽을 비교 분석한다.


---

## 실습 환경 시작

```bash
# 호스트에서 실행
# ELK Stack 시작 (이미 실행 중이면 생략)
cd elk-siem-lab
docker-compose up -d

# 컨테이너 상태 확인
docker-compose ps

# tools 컨테이너 접속 (curl, python3, sigma 사용)

```

## 컴포넌트 설명: KQL (Kibana Query Language)

### KQL 개요

KQL은 Kibana에서 사용하는 검색 쿼리 언어입니다. Kibana Discover, Dashboard 필터에서 사용합니다.

**접속**: http://localhost:5601 → Discover → 검색 바

### KQL 기본 문법

| 문법 | 예시 | 설명 |
|------|------|------|
| 필드:값 | `http.response.status_code: 401` | 정확히 일치 |
| 와일드카드 | `url.path: *login*` | 부분 매칭 |
| AND | `status: 401 AND url.path: *login*` | 두 조건 모두 |
| OR | `tags: sqli OR tags: xss` | 둘 중 하나 |
| NOT | `NOT source.ip: 10.0.1.50` | 제외 |
| 범위 | `http.response.status_code >= 400` | 이상/이하 |
| 존재 | `source.geo: *` | 필드 존재 여부 |

### Discover 사용법

1. **Data View 선택**: `security-*` (전체) 또는 `security-web-*` (웹만)
2. **시간 범위**: 좌측 상단 Time picker → 실습 날짜 선택
3. **검색 바**: KQL 쿼리 입력
4. **필드 선택**: 좌측 Available fields에서 원하는 필드 추가
5. **정렬**: @timestamp 기준 최신순/오래된순

### 실습에서 사용할 주요 KQL 쿼리

```
# SQLi 탐지
tags: sqli

# Brute-force
http.response.status_code: 401 AND url.path: *login*

# XSS
tags: xss

# Path Traversal
tags: path_traversal

# 특정 공격 IP
source.ip: 203.0.113.42

# Sysmon 의심 프로세스
tags: suspicious

# Suricata 알림
event_type: alert
```

## Step 1: KQL 기본 쿼리

### 1-1. Kibana Discover 접속

1. `http://localhost:5601` 접속
2. 좌측 메뉴 -> **Discover** 클릭
3. Data View: `security-all` (`security-*`) 선택
4. 시간 범위를 로그가 포함된 기간으로 조정

### 1-2. KQL 기본 문법

| 문법 | 예시 | 설명 |
|------|------|------|
| 필드 매칭 | `source.ip: "203.0.113.50"` | 특정 값 검색 |
| 와일드카드 | `url.path: *admin*` | 부분 문자열 검색 |
| AND/OR | `A AND B`, `A OR B` | 논리 연산 |
| NOT | `NOT http.response.status_code: 200` | 부정 조건 |
| 범위 | `http.response.status_code >= 400` | 숫자 범위 |
| 존재 여부 | `tags: attack` | 태그/필드 존재 확인 |

### 1-3. 기본 쿼리 실습

전체 공격 태그가 붙은 이벤트를 조회한다:

```
tags: attack
```

정상 트래픽만 조회한다 (공격 태그 제외):

```
NOT tags: attack
```

## Step 2: SQL Injection 탐지

### 2-1. SQLi 패턴 검색

Discover 검색창에 다음 KQL 쿼리를 입력한다:

```
url.path: *union* OR url.path: *select* OR url.path: *1=1*
```

### 2-2. 더 정밀한 SQLi 검색

```
tags: sqli
```

Logstash에서 이미 SQLi 패턴을 태깅하므로, 태그 기반 검색이 더 정확하다.

### 2-3. SQLi 공격 분석

결과에서 다음을 확인한다:

- **공격 출발지 IP**: `source.ip` 필드
- **공격 대상 경로**: `url.path` 필드
- **응답 코드**: `http.response.status_code` 필드
- **MITRE ATT&CK 매핑**: `threat.technique.id` = `T1190` (Exploit Public-Facing Application)

**주요 SQLi 패턴 예시:**

```
/search?q=1' UNION SELECT username,password FROM users--
/login?id=1 OR 1=1--
/product?id=1; DROP TABLE users--
```

### 2-4. SQLi 공격 샘플 로그 (예상 출력)

```json
{
  "source.ip": "203.0.113.42",
  "url.path": "/login?username=admin'%20OR%20'1'%3D'1'--&password=x",
  "url.path_decoded": "/login?username=admin' OR '1'='1'--&password=x",
  "http.response.status_code": 200,
  "http.request.method": "GET",
  "user_agent.original": "sqlmap/1.7.12#stable",
  "tags": ["web", "access", "attack", "sqli"],
  "threat.technique.id": "T1190"
}
```

**분석 포인트:**
- `user_agent.original`: `sqlmap` → 자동화 공격 도구 사용
- `status_code: 200` → 공격이 **성공**했을 가능성 (500이면 에러 기반 SQLi)
- `tags: sqli` → Logstash에서 자동 태깅됨
- `url.path_decoded` vs `url.path` → URL 디코딩 전/후 비교

## Step 3: Brute-force 로그인 공격 탐지

### 3-1. 로그인 실패 검색

```
http.response.status_code: 401 AND url.path: */login*
```

### 3-2. 특정 IP의 반복 로그인 시도 확인

```
url.path: */login* AND http.request.method: POST
```

좌측 필드에서 `source.ip`를 클릭하여 Top values를 확인한다. 특정 IP에서 비정상적으로 많은 로그인 시도가 있는지 분석한다.

### 3-3. 시간대별 분석

1. Discover에서 위 쿼리 적용
2. 히스토그램에서 짧은 시간 내 집중된 요청이 있는지 확인
3. 단시간(수 분 이내)에 수십 건 이상의 401 응답이 집중되면 Brute-force 의심

### 3-4. Brute-force 정량 분석

```bash
# tools 컨테이너 내부에서 실행 (docker exec -it tools bash)
# 분당 요청 수 계산
curl -s 'http://elasticsearch:9200/security-web-*/_search?pretty' \
  -H 'Content-Type: application/json' \
  -d '{
    "size": 0,
    "query": {"bool": {"must": [
      {"term": {"http.response.status_code": 401}},
      {"wildcard": {"url.path": "*login*"}}
    ]}},
    "aggs": {"per_minute": {"date_histogram": {"field": "@timestamp", "fixed_interval": "1m"}}}
  }'
# aggregations.per_minute.buckets 에서 key_as_string(시간)과 doc_count(건수/분) 확인
```

**Brute-force 판정 기준:**
| 지표 | 정상 | Brute-force |
|------|:----:|:-----------:|
| 분당 요청 수 | 1~5회 | **30회 이상** |
| 실패율 | 10~20% | **95% 이상** |
| User-Agent | 다양 | 동일 |
| 요청 간격 | 불규칙 | **2초 간격 일정** |

## Step 4: XSS(Cross-Site Scripting) 탐지

### 4-1. XSS 패턴 검색

```
url.path: *script* OR url.path: *onerror*
```

### 4-2. 태그 기반 검색

```
tags: xss
```

### 4-3. XSS 공격 분석

**주요 XSS 패턴 예시:**

```
/search?q=<script>alert('xss')</script>
/profile?name=test" onerror="alert(1)
/comment?text=<img src=x onerror=alert(document.cookie)>
```

**확인 포인트:**
- `url.path`에 `<script>`, `onerror=`, `onload=`, `javascript:` 포함 여부
- `threat.technique.id` = `T1189` (Drive-by Compromise)

---

## Step 5: Path Traversal 탐지

### 5-1. 디렉토리 탐색 패턴 검색

```
url.path: *..*
```

### 5-2. 태그 기반 검색

```
tags: path_traversal
```

### 5-3. Path Traversal 공격 분석

**주요 패턴 예시:**

```
/files?name=../../../../etc/passwd
/download?file=../../../etc/shadow
/static/%2e%2e/%2e%2e/etc/hosts
```

**확인 포인트:**
- `../` 또는 URL 인코딩된 `%2e%2e%2f` 패턴
- `threat.technique.id` = `T1083` (File and Directory Discovery)
- 응답 코드가 200이면 실제로 파일이 유출되었을 가능성

## Step 6: Sysmon 의심 프로세스 탐지

### 6-1. Data View 변경

Data View를 `security-sysmon`으로 변경한다.

### 6-2. 의심스러운 프로세스 실행 검색

```
tags: suspicious
```

### 6-3. 세부 분류별 검색

**의심스러운 명령 실행:**

```
tags: execution
```

**C2(Command & Control) 통신 의심:**

```
tags: c2
```

**의심스러운 파일 생성:**

```
tags: file_creation
```

### 6-4. 분석 포인트

결과에서 다음 필드를 확인한다:

| 필드 | 분석 내용 |
|------|----------|
| `process.executable` | 실행된 프로세스 (powershell.exe, cmd.exe 등) |
| `process.command_line` | 인코딩된 명령(-enc), 다운로드 명령 등 |
| `process.parent.executable` | 부모 프로세스가 비정상적인지 (예: excel.exe -> cmd.exe) |
| `destination.port` | 알려진 C2 포트 (4444, 5555, 8888) |
| `user.name` | 실행 사용자 권한 |

## Step 7: Suricata IDS 알림 분석

### 7-1. Data View 변경

Data View를 `security-suricata`로 변경한다.

### 7-2. 알림 이벤트 검색

```
event_type: alert
```

### 7-3. 심각도별 분류

```
event_type: alert AND event.severity: 1
```

```
event_type: alert AND event.severity: 2
```

### 7-4. 카테고리별 분류

```
rule.category: *scan*
```

```
rule.category: *trojan* OR rule.category: *malware*
```

### 7-5. 분석 포인트

| 필드 | 분석 내용 |
|------|----------|
| `rule.name` | 탐지 규칙명 (어떤 공격인지) |
| `rule.category` | 공격 분류 (scan, trojan, exploit 등) |
| `event.severity` | 심각도 (1=높음, 2=중간, 3=낮음) |
| `source.ip` | 공격 출발지 |
| `destination.ip` / `destination.port` | 공격 대상 |

## Step 8: 상관 분석 (Cross-Source Correlation)

### 8-1. 통합 Data View 사용

Data View를 `security-all` (`security-*`)로 변경한다.

### 8-2. 동일 IP 교차 확인

웹 공격 로그에서 확인된 공격 IP를 Suricata/Sysmon에서 교차 검색한다.

**웹 로그에서 공격 IP 확인:**

```
tags: attack AND _index: security-web-*
```

좌측 필드에서 `source.ip`의 Top values를 확인하여 공격 IP를 식별한다.

**해당 IP의 Suricata 알림 확인:**

```
source.ip: "203.0.113.42" AND _index: security-suricata-*
```

**해당 IP의 Sysmon 이벤트 확인:**

```
source.ip: "203.0.113.42" AND _index: security-sysmon-*
```

### 8-3. 공격 타임라인 구성

1. 동일 공격 IP로 필터링
2. 시간순 정렬하여 공격 흐름 파악:
   - (1) 포트 스캔 (Suricata) -> (2) 웹 취약점 스캔 (Web) -> (3) SQLi 공격 (Web) -> (4) 악성코드 실행 (Sysmon)

### 8-4. 상관 분석 쿼리 예시

특정 IP에서 발생한 모든 보안 이벤트를 시간순으로 조회:

```
source.ip: "203.0.113.50" AND (tags: attack OR tags: suspicious OR event_type: alert)
```

### 8-5. 상관 분석 실전 예시

공격 IP `203.0.113.42`의 전체 활동을 추적합니다:

```bash
# tools 컨테이너 내부에서 실행 (docker exec -it tools bash)

# 1단계: 웹 공격 확인
curl -s 'http://elasticsearch:9200/security-web-*/_search?pretty' \
  -H 'Content-Type: application/json' \
  -d '{"size":5,"query":{"term":{"source.ip.keyword":"203.0.113.42"}},"sort":[{"@timestamp":"asc"}],"_source":["@timestamp","url.path","http.response.status_code","tags"]}'
# hits.hits[]._source 에서 timestamp, status_code, url.path, tags 확인

# 2단계: IDS 알림 확인
curl -s 'http://elasticsearch:9200/security-suricata-*/_search?pretty' \
  -H 'Content-Type: application/json' \
  -d '{"size":5,"query":{"term":{"source.ip.keyword":"203.0.113.42"}},"sort":[{"@timestamp":"asc"}],"_source":["@timestamp","event_type","rule.name","destination.port"]}'
# hits.hits[]._source 에서 event_type, rule.name, destination.port 확인
```

**Kill Chain 매핑:**
```
09:00~10:00  정찰     → Suricata: "GPL SCAN nmap TCP" (포트 스캔)
11:00~11:30  익스플로잇 → Web: SQLi 공격 (tags: sqli, status 200)
12:00~12:20  익스플로잇 → Web: XSS 공격 (tags: xss)
14:00~14:10  탐색     → Web: Path Traversal (/etc/passwd)
```
