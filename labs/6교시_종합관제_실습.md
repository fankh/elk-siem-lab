## 관제 프로세스 개요

### SOC (Security Operations Center) 관제 흐름

```
탐지 → 트리아지 → 분석 → 격리 → 제거 → 복구 → 교훈
```

### 사용 도구 총정리

| 도구 | 용도 | 접속 |
|------|------|------|
| **Kibana Dashboard** | 실시간 모니터링, Error Spike 확인 | http://localhost:5601 |
| **Kibana Discover** | 드릴다운 분석, KQL 검색 | Dashboard에서 클릭 |
| **Kibana Maps** | 공격 출발지 GeoIP 시각화 | Analytics → Maps |
| **Dev Tools** | ES API 직접 실행 | Management → Dev Tools |
| **Sigma Rules** | 탐지 룰 작성/변환 | `sigma convert` CLI |
| **관제 보고서** | 사고 대응 문서화 | 관제_보고서_양식.docx |

### 경보 분류 기준 (Triage)

| 등급 | 기준 | 대응 시간 | 예시 |
|------|------|:--------:|------|
| **Critical** | 공격 성공 + 데이터 유출 | 즉시 | SQLi로 DB 데이터 추출 성공 |
| **High** | 공격 시도 + 취약점 존재 | 30분 이내 | Brute-force 로그인 성공 |
| **Medium** | 공격 시도 + 차단됨 | 4시간 이내 | SQLi 시도 → 403 차단 |
| **Low** | 스캔/정찰 활동 | 일일 리뷰 | 포트 스캔, 디렉토리 탐색 |

### Cyber Kill Chain 7단계

| 단계 | 설명 | 로그 소스 |
|------|------|----------|
| 1. 정찰 | 포트 스캔, 디렉토리 탐색 | Suricata, Web |
| 2. 무기화 | 공격 도구 준비 | - |
| 3. 전달 | 공격 페이로드 전송 | Web (SQLi, XSS) |
| 4. 익스플로잇 | 취약점 실행 | Web (200 응답) |
| 5. 설치 | 악성코드/웹쉘 설치 | Sysmon (파일 생성) |
| 6. C2 | 명령제어 통신 | Sysmon (네트워크), Suricata |
| 7. 목표 달성 | 데이터 유출/파괴 | DB, Web |

### 위험도 산정 공식

```
위험도 = 위협 (1~5) × 취약성 (1~5) × 자산 가치 (1~5)
```

| 점수 범위 | 등급 | 대응 |
|:---------:|:----:|------|
| 75~125 | Critical | 즉시 대응, 경영진 보고 |
| 40~74 | High | 4시간 이내 대응 |
| 15~39 | Medium | 24시간 이내 대응 |
| 1~14 | Low | 모니터링 |

내용: 외부 IP에서 웹 서버에 대한 다수의 비정상 요청 탐지

  - HTTP 4xx/5xx 에러율 급증

## Step 5: Kill-chain 매핑

식별된 공격 활동을 **Cyber Kill Chain** 단계에 매핑한다.

### Cyber Kill Chain 단계

```
1. 정찰 (Reconnaissance)
   └─ 포트 스캔, 디렉토리 스캔, 기술 스택 파악

2. 무기화 (Weaponization)
   └─ 공격 도구/페이로드 준비 (로그에서 직접 관찰 불가)

3. 전달 (Delivery)
   └─ 공격 요청 전송 (HTTP 요청)

4. 익스플로잇 (Exploitation)
   └─ SQLi, XSS, Path Traversal 실행

5. 설치 (Installation)
   └─ 웹쉘 업로드, 백도어 설치

6. C2 통신 (Command & Control)
   └─ 외부 서버와 통신 (Sysmon/Suricata 로그 확인)

7. 목표 달성 (Actions on Objectives)
   └─ 데이터 유출, 시스템 파괴
```

### 매핑 실습

Discover에서 각 단계에 해당하는 로그를 검색한다:

**정찰 단계 확인:**
```
response: 404 AND url.path: (*admin* OR *backup* OR *config* OR *.env* OR *wp-*)
```

**익스플로잇 단계 확인:**
```
url.query: (*UNION* OR *SELECT* OR *<script>* OR *../*)
```

**C2 통신 확인 (Sysmon 로그):**
```
event.category: "network_connection" AND destination.port: (4444 OR 8080 OR 1234)
```

## Step 6: 오탐(False Positive) 식별

탐지된 이벤트 중 실제 공격이 아닌 정상 트래픽(오탐)을 식별한다.

### 오탐 판별 기준

| 구분 | 실제 공격 | 오탐 (False Positive) |
|------|-----------|----------------------|
| 요청 패턴 | 체계적, 반복적 | 단발성, 무작위 |
| User-Agent | 자동화 도구, 스크립트 | 일반 브라우저 |
| 페이로드 | 실행 가능한 공격 코드 | 검색어, 정상 파라미터 |
| 시간 패턴 | 짧은 간격 집중 | 불규칙, 간헐적 |
| 응답 코드 | 다수의 4xx | 혼합 (2xx + 4xx) |

### 오탐 사례 분석

다음 요청들을 확인하고 공격 vs 오탐을 판별하시오:

1. `GET /search?q=select+a+nice+restaurant` - 정상 검색어에 "select" 포함
2. `GET /products?category=shoes&sort=price` - 정상 파라미터
3. `GET /api/data?query=SELECT * FROM users` - SQL 쿼리 직접 전송
4. `GET /page?content=<b>bold text</b>` - HTML 태그 포함

### 화이트리스트 작성

오탐으로 확인된 패턴을 화이트리스트에 등록한다:

```yaml
# whitelist.yml
whitelist_rules:
  - name: "정상 검색 쿼리"
    field: url.path
    value: "/search"
    reason: "검색 기능의 정상 사용"

  - name: "내부 모니터링"
    field: source.ip
    value: "10.0.0.0/8"
    reason: "내부 헬스체크 트래픽"
```

### 6-4. 오탐 판별 정답 및 해설

| # | 요청 | 판정 | 근거 |
|---|------|:----:|------|
| 1 | `/search?q=SELECT+TOP+10+products` | **오탐 (FP)** | `SELECT TOP`은 SQL 키워드이지만, 검색어에서 제품명으로 사용. User-Agent가 일반 브라우저이고 출발지가 내부 IP |
| 2 | `/login` POST → 401 (3회) | **오탐 (FP)** | 3회 실패는 사용자 비밀번호 분실 가능성. 임계값(20회) 미만 |
| 3 | `/api/users?id=1 UNION SELECT password FROM users--` | **공격 (TP)** | UNION SELECT + 비밀번호 테이블 직접 조회. sqlmap User-Agent |
| 4 | `/health?check=../config/status` | **오탐 (FP)** | 내부 헬스체크 경로. `../`가 포함되지만 정기 점검 트래픽 (동일 IP에서 매분 반복) |

**오탐 제거 화이트리스트 예시:**
```yaml
# Sigma Rule에 falsepositives 추가
falsepositives:
    - "Internal health check from 10.0.1.50 to /health endpoint"
    - "Product search containing SQL keywords"
```

## Step 7: 위험도 산정

식별된 위협에 대해 정량적 위험도를 산정한다.

### 위험도 공식

```
위험도(Risk) = 위협(Threat) x 취약성(Vulnerability) x 자산 가치(Asset Value)
```

### 평가 기준 (1~5점)

**위협 (Threat):**

| 점수 | 기준 |
|------|------|
| 5 | 활성 익스플로잇, 자동화 공격 |
| 4 | 수동 공격 시도, 알려진 취약점 대상 |
| 3 | 정찰 활동, 스캐닝 |
| 2 | 비표적 자동 스캔 |
| 1 | 봇 트래픽, 단순 노이즈 |

**취약성 (Vulnerability):**

| 점수 | 기준 |
|------|------|
| 5 | 패치 미적용, 알려진 CVE 존재 |
| 4 | 설정 미흡, 기본 계정 사용 |
| 3 | 일부 보안 조치 적용 |
| 2 | 대부분 보안 조치 적용 |
| 1 | 완전한 보안 조치 적용 |

**자산 가치 (Asset Value):**

| 점수 | 기준 |
|------|------|
| 5 | 핵심 DB, 개인정보 포함 |
| 4 | 업무 핵심 시스템 |
| 3 | 일반 업무 시스템 |
| 2 | 개발/테스트 환경 |
| 1 | 공개 정보만 포함 |

### 위험도 산정 실습

| 공격 유형 | 위협 | 취약성 | 자산 가치 | 위험도 (총점) | 등급 |
|-----------|------|--------|-----------|--------------|------|
| SQL Injection | | | | | |
| XSS | | | | | |
| Brute Force | | | | | |
| Path Traversal | | | | | |

**위험도 등급:**
- 75~125: Critical (즉시 대응)
- 36~74: High (긴급 대응)
- 12~35: Medium (계획 대응)
- 1~11: Low (모니터링)
