## Step 2: Elasticsearch 설치 및 시작

### docker-compose.yml 확인

```yaml
elasticsearch:
  image: docker.elastic.co/elasticsearch/elasticsearch:8.12.0
  container_name: elasticsearch
  environment:
    - discovery.type=single-node       # 단일 노드 (실습용)
    - xpack.security.enabled=false     # 보안 비활성화 (실습용)
    - ES_JAVA_OPTS=-Xms2g -Xmx2g      # JVM 힙 메모리 2GB
    - cluster.name=siem-lab
  ports:
    - "9200:9200"                      # REST API
  volumes:
    - es-data:/usr/share/elasticsearch/data  # 데이터 영속화
```

### 핵심 설정 설명

| 설정 | 값 | 설명 |
|------|-----|------|
| `discovery.type` | `single-node` | 클러스터 없이 단독 실행 |
| `xpack.security.enabled` | `false` | 인증 없이 접근 (실습 전용) |
| `ES_JAVA_OPTS` | `-Xms2g -Xmx2g` | 최소/최대 힙 2GB (동일하게 설정) |
| 포트 `9200` | REST API | `curl http://localhost:9200` 으로 접근 |

### 시작 및 확인

```bash
# Elasticsearch만 먼저 시작 (호스트에서 실행)
docker-compose up -d elasticsearch

# 상태 확인 (~60초 대기)
# tools 컨테이너에서:
docker exec -it tools curl -s http://elasticsearch:9200/_cluster/health?pretty
# Windows 호스트에서: curl.exe -s http://localhost:9200/_cluster/health?pretty
# macOS/Linux 호스트에서: curl -s http://localhost:9200/_cluster/health?pretty
# 기대: "status" : "yellow" (single-node이므로 yellow 정상)

# 버전 확인
docker exec -it tools curl -s http://elasticsearch:9200?pretty
# Windows 호스트에서: curl.exe -s http://localhost:9200?pretty
# 기대: "number" : "8.12.0", "cluster_name" : "siem-lab"
```

## Step 3: Kibana 설치 및 시작

### docker-compose.yml 확인

```yaml
kibana:
  image: docker.elastic.co/kibana/kibana:8.12.0
  container_name: kibana
  environment:
    - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    - xpack.security.enabled=false
  ports:
    - "5601:5601"                      # Kibana UI
  depends_on:
    elasticsearch:
      condition: service_healthy       # ES가 healthy 된 후 시작
```

### 핵심 설정 설명

| 설정 | 값 | 설명 |
|------|-----|------|
| `ELASTICSEARCH_HOSTS` | `http://elasticsearch:9200` | Docker 내부 네트워크로 ES 연결 |
| 포트 `5601` | Kibana UI | `http://localhost:5601` 로 접속 |
| `depends_on` | `service_healthy` | ES 헬스체크 통과 후 시작 |

### 시작 및 확인

```bash
# Kibana 시작 (호스트에서 실행)
docker-compose up -d kibana

# 상태 확인 (~30초 대기)
# tools 컨테이너에서:
docker exec -it tools curl -s http://kibana:5601/api/status | head -5
# Windows 호스트에서: curl.exe -s http://localhost:5601/api/status | head -5
# 기대: "level" : "available"

# 브라우저 접속
# http://localhost:5601
```

## Step 4: Logstash 설치 및 시작

### docker-compose.yml 확인

```yaml
logstash:
  image: docker.elastic.co/logstash/logstash:8.12.0
  container_name: logstash
  volumes:
    - ./logstash/pipeline:/usr/share/logstash/pipeline:ro  # 파이프라인 설정
    - ./sample-logs:/var/log/sample:ro                      # 샘플 로그
  ports:
    - "5044:5044"                      # Beats 입력
  environment:
    - LS_JAVA_OPTS=-Xms512m -Xmx512m  # JVM 512MB
  depends_on:
    elasticsearch:
      condition: service_healthy
```

### 핵심 설정 설명

| 설정 | 값 | 설명 |
|------|-----|------|
| `./logstash/pipeline` | `:ro` 마운트 | 3개 conf 파일 자동 로드 |
| `./sample-logs` | `:ro` 마운트 | 샘플 로그 읽기 전용 |
| 포트 `5044` | Beats 입력 | Filebeat → Logstash 수신 |
| `LS_JAVA_OPTS` | `-Xms512m -Xmx512m` | Logstash 메모리 512MB |

### 파이프라인 파일 확인

```bash
ls logstash/pipeline/
# web-access.conf     ← 웹 로그 파싱 (Grok + GeoIP + 공격 태깅)
# sysmon.conf         ← Sysmon ECS 매핑 + 의심 프로세스 태깅
# suricata.conf       ← Suricata IDS 파싱 + 공격 분류
```

### 시작 및 확인

```bash
# Logstash 시작 (호스트에서 실행)
docker-compose up -d logstash

# 파이프라인 시작 확인 (~30초 대기, 호스트에서 실행)
docker logs logstash 2>&1 | grep "Pipeline started"
# 기대: Pipeline started {"pipeline.id"=>"main"}

# 인덱스 생성 확인 (~90초 대기, 로그 처리 후)
# tools 컨테이너에서:
docker exec -it tools curl -s http://elasticsearch:9200/_cat/indices/security-*?v
# Windows 호스트에서: curl.exe -s http://localhost:9200/_cat/indices/security-*?v
# 기대: security-web-*, security-sysmon-*, security-suricata-* 3개 인덱스
```

## Step 5: Filebeat 설치 및 시작

### docker-compose.yml 확인

```yaml
filebeat:
  image: docker.elastic.co/beats/filebeat:8.12.0
  container_name: filebeat
  user: root                           # 로그 파일 읽기 권한
  volumes:
    - ./filebeat/filebeat.yml:/usr/share/filebeat/filebeat.yml:ro
    - ./sample-logs:/var/log/sample:ro
  depends_on:
    elasticsearch:
      condition: service_healthy
```

### filebeat.yml 설정

```yaml
filebeat.inputs:
  - type: log
    paths: ["/var/log/sample/web-access.log"]
    tags: ["web", "access"]            # 태그로 라우팅

  - type: log
    paths: ["/var/log/sample/sysmon.json"]
    tags: ["sysmon", "endpoint"]
    json.keys_under_root: true         # JSON 필드를 최상위로

  - type: log
    paths: ["/var/log/sample/suricata-eve.json"]
    tags: ["suricata", "ids"]
    json.keys_under_root: true

output.logstash:
  hosts: ["logstash:5044"]             # Logstash로 전송
```

### 시작 및 확인

```bash
# Filebeat 시작
docker-compose up -d filebeat

# 로그 확인
docker logs filebeat 2>&1 | tail -5
# 기대: Harvester started for paths: [/var/log/sample/web-access.log] 등
```

> **참고**: Filebeat 로그에 `connection refused (logstash:5044)` 에러가 출력될 수 있습니다. 본 실습에서 Logstash는 `file` input으로 로그를 직접 읽으므로 Beats 입력(5044)을 사용하지 않습니다. 이 에러는 무시해도 되며, 데이터는 Logstash를 통해 정상 적재됩니다.

## Step 6: Sigma CLI 설치

Sigma Rule을 Elasticsearch 쿼리로 변환하는 CLI 도구입니다.

> **참고**: `tools` 컨테이너에는 `sigma-cli`가 사전 설치되어 있습니다. 호스트에서 직접 사용하려면 아래 수동 설치를 진행하세요.

**tools 컨테이너에서 사용 (권장):**
```bash
docker exec -it tools bash
sigma version
# 기대: pySigma version x.x.x

# 변환 테스트
sigma convert -t lucene --without-pipeline /lab/sigma-rules/sqli-detection.yml
# 기대: Elasticsearch bool 쿼리 JSON 출력
```

**호스트에 수동 설치 (선택):**
```bash
# Python 가상환경 (권장)
python3 -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate

# sigma-cli + ES 백엔드 설치
pip install sigma-cli pySigma-backend-elasticsearch

# 설치 확인
sigma version

# 변환 테스트 (호스트 경로)
sigma convert -t lucene --without-pipeline sigma-rules/sqli-detection.yml
```

### Sigma Rule 파일 확인

```bash

# xss-detection.yml         ← XSS (T1189)
```

## Step 8: 환경 검증 체크리스트

모든 서비스가 정상인지 한 번에 확인합니다.

```bash
# tools 컨테이너 내부에서 실행 (docker exec -it tools bash)
# Windows 호스트에서는 curl → curl.exe, elasticsearch/kibana → localhost 로 변경

echo "=== 환경 검증 ==="

# 1. Elasticsearch
echo "[1] Elasticsearch"
curl -s http://elasticsearch:9200/_cluster/health?pretty | head -5

# 2. Kibana
echo "[2] Kibana"
curl -s http://kibana:5601/api/status | head -3

# 3. Logstash Pipeline (호스트에서 실행: docker logs logstash 2>&1 | grep "Pipeline started")

# 4. 인덱스 문서 수
echo "[4] Indices"
for idx in web sysmon suricata; do

done

# 5. Sigma CLI
echo "[5] Sigma CLI"
sigma version 2>/dev/null && echo "  installed" || echo "  not installed (pip install sigma-cli)"

echo ""
echo "=== 검증 완료 ==="
```

## 트러블슈팅

### Elasticsearch가 시작되지 않는 경우

```bash
# 에러 로그 확인
docker logs elasticsearch 2>&1 | tail -20

# vm.max_map_count 오류 (Linux)
sudo sysctl -w vm.max_map_count=262144
echo "vm.max_map_count=262144" | sudo tee -a /etc/sysctl.conf

# 메모리 부족 → .env 수정
ES_JAVA_OPTS=-Xms1g -Xmx1g
```

### Logstash 파이프라인 에러

```bash
# 에러 확인
docker logs logstash 2>&1 | grep ERROR

# 설정 문법 검증
docker exec logstash logstash --config.test_and_exit -f /usr/share/logstash/pipeline/
```

### Kibana가 접속되지 않는 경우

```bash
# ES 연결 확인
docker logs kibana 2>&1 | grep -i "error\|unable"

# Kibana 재시작
docker-compose restart kibana
```

### 인덱스가 생성되지 않는 경우

```bash
# Logstash가 로그를 읽고 있는지 확인
docker logs logstash 2>&1 | grep "Pipeline started"

# sample-logs 파일 존재 확인
docker exec logstash ls /var/log/sample/
# 기대: web-access.log, sysmon.json, suricata-eve.json

# 데이터 초기화 후 재시작
docker-compose down -v
docker-compose up -d
```

### 포트 충돌

```bash
# 사용 중인 포트 확인
lsof -i :9200   # ES
lsof -i :5601   # Kibana
lsof -i :5044   # Logstash

# .env 또는 docker-compose.yml에서 포트 변경
```
