# 실습 1: 로그 수집 파이프라인

## 목표

Filebeat와 Logstash를 활용하여 이기종 로그(웹 액세스, Sysmon, Suricata) 수집 파이프라인을 구성하고, Elasticsearch에 정상적으로 인덱싱되는 것을 확인한다.

---

## 실습 환경 시작

```bash
# 호스트에서 실행
# 1. 저장소 클론 (최초 1회)
git clone https://github.com/fankh/elk-siem-lab.git
cd elk-siem-lab

# 2. ELK Stack 전체 시작 (ES, Kibana, Logstash, Filebeat, tools)
docker-compose up -d

# 3. 컨테이너 상태 확인
docker-compose ps
# 기대: elasticsearch, kibana, logstash, filebeat, tools 모두 Up

# 4. ES 상태 확인 (~60초 대기)
docker exec -it tools curl http://elasticsearch:9200/_cluster/health?pretty
# Windows 호스트에서: curl.exe http://localhost:9200/_cluster/health?pretty
# 기대: "status" : "yellow"
```

> **참고**: 첫 실행 시 Docker 이미지 다운로드에 ~5분, ES 초기화에 ~60초, Logstash 로그 처리에 ~90초 소요

> **CLI 실행 환경**
> | 환경 | 명령 |
> |------|------|
> | **tools 컨테이너 (권장)** | `docker exec -it tools bash` → 내부에서 curl, python3, sigma 사용 |
> | **macOS/Linux** | 호스트 터미널에서 `curl localhost:9200/...` |
> | **Windows PowerShell** | `curl.exe localhost:9200/...` 또는 브라우저 직접 접속 |
> | **브라우저** | http://localhost:9200 (ES) / http://localhost:5601 (Kibana) |

## 컴포넌트 설치: Filebeat & Logstash

### Filebeat 설치

Filebeat는 경량 로그 수집 에이전트입니다. 본 실습에서는 Docker로 실행됩니다.

**Docker (본 실습)**
```bash
# docker-compose.yml에 포함됨 (자동 실행)
docker-compose up -d filebeat
```

**수동 설치 (참고용)**
```bash
# Linux
curl -L -O https://artifacts.elastic.co/downloads/beats/filebeat/filebeat-8.12.0-linux-x86_64.tar.gz
tar xzvf filebeat-8.12.0-linux-x86_64.tar.gz

# macOS
brew install elastic/tap/filebeat-full

# Windows - elastic.co에서 ZIP 다운로드
```

**핵심 설정 (filebeat/filebeat.yml)**
```yaml
filebeat.inputs:
  - type: log
    paths: ["/var/log/sample/web-access.log"]
    tags: ["web", "access"]          # 태그로 Logstash에서 라우팅
  - type: log
    paths: ["/var/log/sample/sysmon.json"]
    tags: ["sysmon", "endpoint"]
    json.keys_under_root: true       # JSON 필드를 최상위로
  - type: log
    paths: ["/var/log/sample/suricata-eve.json"]
    tags: ["suricata", "ids"]
    json.keys_under_root: true

output.logstash:
  hosts: ["logstash:5044"]           # Logstash 포트 5044로 전송
```

| 설정 | 설명 |
|------|------|
| `type: log` | 파일 기반 로그 수집 |
| `tags` | Logstash에서 이벤트 라우팅에 사용 |
| `json.keys_under_root` | JSON 로그의 필드를 최상위로 추출 |
| `output.logstash` | Logstash Beats 입력(5044)으로 전송 |

### Logstash 설치

Logstash는 로그 파싱, 변환, 라우팅을 수행하는 파이프라인 엔진입니다.

**Docker (본 실습)**
```bash
docker-compose up -d logstash
```

**수동 설치 (참고용)**
```bash
# Linux
wget https://artifacts.elastic.co/downloads/logstash/logstash-8.12.0-linux-x86_64.tar.gz
tar xzvf logstash-8.12.0-linux-x86_64.tar.gz

# macOS
brew install elastic/tap/logstash-full
```

**파이프라인 구조 (Input → Filter → Output)**
```
logstash/pipeline/
├── web-access.conf     # 웹 로그: Grok 파싱 + GeoIP + 공격 태깅
├── sysmon.conf         # Sysmon: ECS 매핑 + 의심 프로세스 태깅
└── suricata.conf       # Suricata: IDS 알림 파싱 + 공격 분류
```

| 파이프라인 | Input | Filter | Output Index |
|-----------|-------|--------|-------------|
| web-access.conf | file (web-access.log) | grok, date, geoip, useragent, urldecode | security-web-* |
| sysmon.conf | file (sysmon.json) | date, mutate (ECS rename) | security-sysmon-* |
| suricata.conf | file (suricata-eve.json) | date, mutate (ECS rename), geoip | security-suricata-* |

> **⚠️ 아키텍처 참고**: 본 실습에서 Logstash는 `file` input 플러그인으로 샘플 로그 파일을 **직접 읽습니다** (`/var/log/sample/` 마운트). Logstash 파이프라인에 `beats` input이 없으므로 **Filebeat → Logstash(5044) 연결은 사용되지 않습니다**. Filebeat 로그에 `connection refused (logstash:5044)` 에러가 반복 출력되지만, 이는 정상입니다 — 데이터는 Logstash가 파일을 직접 읽어 ES로 전송합니다.
>
> ```
> 실제 데이터 흐름:  sample-logs → (volume mount) → Logstash file input → ES
> Filebeat (미사용):  Filebeat → logstash:5044 (연결 실패, 무시 가능)
> ```

**태그 기반 라우팅**: 각 conf 파일은 `if "web" in [tags]` 조건으로 해당 로그만 처리합니다.

### 왜 Filebeat → Logstash → ES 구조인가?

Filebeat에서 ES로 직접 전송할 수도 있지만, 본 실습에서는 Logstash를 경유합니다.

| 항목 | Filebeat → ES (직접) | Filebeat → Logstash → ES |
|------|:-------------------:|:------------------------:|
| **파싱** | Ingest Pipeline만 (제한적) | Grok, JSON, CSV 등 풍부한 필터 |
| **변환/보강** | 기본 프로세서만 | GeoIP, UserAgent, urldecode |
| **공격 태깅** | 불가 | regex 조건부 태깅 (sqli, xss 등) |
| **라우팅** | 1 input → 1 index | 태그 기반 다중 인덱스 분기 |
| **버퍼링** | 없음 (ES 장애 시 유실) | Persistent Queue로 유실 방지 |
| **ES 부하** | 파싱이 ES에 집중 | Logstash가 파싱, ES는 저장만 |

**본 실습에서 Logstash가 필수인 이유:**
- Grok으로 Apache Combined 로그를 필드 단위로 파싱
- `urldecode` 후 regex로 SQLi/XSS/Path Traversal **공격 태깅**
- GeoIP로 외부 IP → 국가/도시 보강
- 태그 기반으로 `security-web-*` / `security-sysmon-*` / `security-suricata-*` **인덱스 분기**

**실무 아키텍처 비교:**

```
소규모 (파싱 불필요):  Filebeat → ES 직접
중규모 (파싱 필요):    Filebeat → Logstash → ES  ← 본 실습
대규모 (버퍼 필요):    Filebeat → Kafka → Logstash → ES
```

- **Output**: `security-suricata-YYYY.MM.dd` 인덱스로 전송

### 2-5. web-access.conf 필터 체인 상세

웹 로그의 전체 처리 흐름입니다:

```ruby
filter {
  if "web" in [tags] {
    # 1단계: Grok 파싱 (원본 로그 → 구조화 필드)
    grok {
      match => {
        "message" => '%{IPORHOST:source.ip} - %{DATA:user.name} \[%{HTTPDATE:timestamp}\] "%{WORD:http.request.method} %{URIPATHPARAM:url.path} HTTP/%{NUMBER}" %{NUMBER:http.response.status_code:int} %{NUMBER:http.response.body.bytes:int} "%{DATA:http.request.referrer}" "%{DATA:user_agent.original}"'
      }
    }

    # 2단계: 타임스탬프 변환
    date { match => ["timestamp", "dd/MMM/yyyy:HH:mm:ss Z"] }

    # 3단계: GeoIP 보강 (외부 IP → 국가/도시)
    geoip { source => "source.ip" target => "source.geo" }

    # 4단계: User-Agent 파싱
    useragent { source => "user_agent.original" target => "user_agent" }

    # 5단계: URL 디코딩 (공격 탐지를 위해)
    mutate { copy => { "url.path" => "url.path_decoded" } }
    urldecode { field => "url.path_decoded" }

    # 6단계: 공격 패턴 매칭 → 태그 추가
    if [url.path_decoded] =~ /(?i)(union.+select|or.+1.+=.+1|drop\s+table|sleep\(|'--)/
    { mutate { add_tag => ["attack", "sqli"] } }

    if [url.path_decoded] =~ /(?i)(<script|javascript:|onerror\s*=)/
    { mutate { add_tag => ["attack", "xss"] } }

    if [url.path_decoded] =~ /(\.\.\/|\.\.%2[fF])/
    { mutate { add_tag => ["attack", "path_traversal"] } }
  }
}
```

**처리 흐름 요약:**
```
원본 로그 → Grok(필드 추출) → Date(시간 변환) → GeoIP(위치 보강) → UA(브라우저 파싱) → URLDecode → 공격 태깅 → ES 저장
```

## Step 5: Kibana에서 인덱스 패턴 생성

### 5-1. Kibana 접속

브라우저에서 `http://localhost:5601` 접속

### 5-2. Data View (인덱스 패턴) 생성

1. 좌측 메뉴 -> **Stack Management** -> **Data Views** 클릭
2. **Create data view** 클릭
3. 다음 3개의 Data View를 생성:

| Name | Index pattern | Timestamp field |
|------|---------------|-----------------|
| security-web | `security-web-*` | `@timestamp` |
| security-sysmon | `security-sysmon-*` | `@timestamp` |
| security-suricata | `security-suricata-*` | `@timestamp` |

4. 통합 조회용 Data View 추가 생성:

| Name | Index pattern | Timestamp field |
|------|---------------|-----------------|
| security-all | `security-*` | `@timestamp` |

### 5-3. Discover에서 확인

1. 좌측 메뉴 -> **Discover** 클릭
2. 좌측 상단 Data View 드롭다운에서 `security-all` 선택
3. 시간 범위를 적절히 조정하여 로그가 표시되는지 확인

## 트러블슈팅

### 인덱스가 생성되지 않는 경우

```bash
# 호스트에서 실행
# Logstash 에러 확인
docker logs logstash 2>&1 | grep ERROR

# 파이프라인 실행 확인
docker logs logstash 2>&1 | grep "Pipeline started"
# 기대: Pipeline started {"pipeline.id"=>"main"}

# 샘플 로그 파일 마운트 확인
docker exec logstash ls /var/log/sample/
# 기대: web-access.log, sysmon.json, suricata-eve.json
```

### Filebeat → Logstash 연결 실패 (`connection refused: logstash:5044`)

> **이 에러는 정상입니다.** 본 실습의 Logstash 파이프라인은 `file` input으로 로그를 직접 읽으며, `beats` input(5044)을 사용하지 않습니다. Filebeat의 `connection refused` 에러는 무시해도 됩니다. 데이터가 ES에 정상 적재되었는지 확인하세요:

```bash
# 데이터 적재 확인 (이것이 정상이면 Filebeat 에러 무시 가능)
docker exec tools curl -s 'http://elasticsearch:9200/_cat/indices/security-*?v'
# 기대: security-web-* (617건), security-sysmon-* (229건), security-suricata-* (149건)
```

### 데이터 초기화 후 재시작

```bash
# 호스트에서 실행
docker-compose down -v    # 볼륨 삭제
docker-compose up -d      # 재시작
# ~90초 대기 후 인덱스 확인
```
