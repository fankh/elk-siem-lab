## 컴포넌트 설명: Grok & ECS

### Grok 패턴 엔진

Grok은 Logstash의 핵심 파싱 엔진으로, 정규표현식 패턴을 이름으로 매핑합니다.

**설치**: Logstash에 내장 (별도 설치 불필요)

**주요 내장 패턴**

| 패턴 | 설명 | 매칭 예시 |
|------|------|----------|
| `%{IPORHOST}` | IP 또는 호스트명 | `192.168.1.1`, `example.com` |
| `%{HTTPDATE}` | Apache 날짜 형식 | `26/Mar/2026:10:00:00 +0900` |
| `%{WORD}` | 단일 단어 | `GET`, `POST` |
| `%{URIPATHPARAM}` | URI 경로 + 파라미터 | `/api/users?id=1` |
| `%{NUMBER}` | 숫자 | `200`, `404` |
| `%{DATA}` | 임의 문자 (비탐욕) | `Mozilla/5.0...` |
| `%{GREEDYDATA}` | 임의 문자 (탐욕) | 나머지 전체 |

**Grok Debugger**: Kibana → Dev Tools → Grok Debugger 에서 패턴 테스트 가능

### ECS (Elastic Common Schema)

ECS는 Elasticsearch의 필드 표준화 규칙입니다. 서로 다른 로그 소스의 필드를 통일된 이름으로 매핑합니다.

| 카테고리 | ECS 필드 | 설명 |
|---------|---------|------|
| 네트워크 | `source.ip` | 출발지 IP |
| 네트워크 | `destination.ip` | 목적지 IP |
| HTTP | `url.path` | 요청 URL 경로 |
| HTTP | `http.response.status_code` | HTTP 상태 코드 |
| 프로세스 | `process.executable` | 실행 파일 경로 |
| 프로세스 | `process.command_line` | 명령행 인자 |
| 탐지 | `rule.name` | 탐지 룰 이름 |
| 위치 | `source.geo.country_name` | GeoIP 국가 |

### GeoIP 플러그인

외부 IP를 국가/도시 정보로 변환합니다. Logstash에 내장된 MaxMind GeoLite2 데이터베이스를 사용합니다.

```ruby
geoip {
  source => "source.ip"
  target => "source.geo"
}
# 결과: source.geo.country_name, source.geo.city_name, source.geo.location
```

## Step 2: Grok 패턴 분석

### 2-1. 웹 액세스 로그 Grok 패턴

`logstash/pipeline/web-access.conf`의 grok match 구문을 분석한다.

```ruby
grok {
  match => {
    "message" => '%{IPORHOST:source.ip} - %{DATA:user.name} \[%{HTTPDATE:timestamp}\] "%{WORD:http.request.method} %{URIPATHPARAM:url.path} HTTP/%{NUMBER:http.version}" %{NUMBER:http.response.status_code:int} %{NUMBER:http.response.body.bytes:int} "%{DATA:http.request.referrer}" "%{DATA:user_agent.original}"'
  }
}
```

### 2-2. Grok 패턴 분해

| Grok 패턴 | 매핑 필드 | 설명 |
|-----------|----------|------|
| `%{IPORHOST}` | `source.ip` | 클라이언트 IP 주소 |
| `%{DATA}` | `user.name` | 인증된 사용자명 |
| `%{HTTPDATE}` | `timestamp` | 요청 시각 (후에 @timestamp로 변환) |
| `%{WORD}` | `http.request.method` | HTTP 메서드 (GET, POST 등) |
| `%{URIPATHPARAM}` | `url.path` | 요청 URI 경로 |
| `%{NUMBER}` | `http.version` | HTTP 버전 |
| `%{NUMBER:int}` | `http.response.status_code` | HTTP 응답 코드 (정수 변환) |
| `%{NUMBER:int}` | `http.response.body.bytes` | 응답 바이트 수 (정수 변환) |
| `%{DATA}` | `http.request.referrer` | 리퍼러 URL |
| `%{DATA}` | `user_agent.original` | User-Agent 문자열 |

### 2-3. Grok 파싱 단계별 예시

원본 로그 한 줄을 Grok으로 파싱하는 과정입니다:

**원본:**
```
203.0.113.42 - - [26/Mar/2026:11:30:15 +0900] "GET /login?username=admin'%20OR%20'1'%3D'1'-- HTTP/1.1" 200 3842 "-" "sqlmap/1.7.12"
```

**파싱 결과:**

| Grok 패턴 | 추출 필드 | 값 |
|-----------|----------|-----|
| `%{IPORHOST}` | `source.ip` | `203.0.113.42` |
| `%{DATA}` | `user.name` | `-` |
| `%{HTTPDATE}` | `timestamp` | `26/Mar/2026:11:30:15 +0900` |
| `%{WORD}` | `http.request.method` | `GET` |
| `%{URIPATHPARAM}` | `url.path` | `/login?username=admin'%20OR%20'1'%3D'1'--` |
| `%{NUMBER:int}` | `http.response.status_code` | `200` |
| `%{NUMBER:int}` | `http.response.body.bytes` | `3842` |
| `%{DATA}` | `http.request.referrer` | `-` |
| `%{DATA}` | `user_agent.original` | `sqlmap/1.7.12` |

**후처리:**
- `Date` 필터: `timestamp` → `@timestamp` (ISO8601 변환)
- `GeoIP`: `203.0.113.42` → `source.geo.country_name: "Test"` (RFC5737 테스트 IP)
- `URLDecode`: `%20` → ` `, `%3D` → `=` → 공격 패턴 매칭 → `tags: ["attack", "sqli"]`

### 2-4. Grok Debugger로 테스트

1. Kibana 좌측 메뉴 -> **Dev Tools** -> **Grok Debugger**
2. Sample Data에 원본 로그 한 줄 붙여넣기
3. Grok Pattern에 위 패턴 입력
4. **Simulate** 클릭하여 파싱 결과 확인

## Step 3: ECS 필드 확인 - 웹 로그

### 3-1. Kibana Discover에서 필드 확인

1. Data View: `security-web` 선택
2. 좌측 필드 목록에서 다음 필드를 찾아 확인:

```
source.ip
url.path
http.request.method
http.response.status_code
http.response.body.bytes
user.name
user_agent.original
```

### 3-2. 필드별 상위 값 확인

좌측 필드 목록에서 각 필드를 클릭하면 상위 값(Top values)을 확인할 수 있다.

**확인할 항목:**
- `http.request.method`: GET, POST 등의 분포
- `http.response.status_code`: 200, 401, 403, 404, 500 등의 분포
- `source.ip`: 어떤 IP에서 접근이 많은지

### 3-3. ES API로 필드 분석

```bash
# tools 컨테이너 내부에서 실행 (docker exec -it tools bash)

# HTTP 상태 코드 분포
curl -s 'http://elasticsearch:9200/security-web-*/_search?pretty' \
  -H 'Content-Type: application/json' \
  -d '{"size":0,"aggs":{"status_codes":{"terms":{"field":"http.response.status_code","size":10}}}}'
# aggregations.status_codes.buckets 에서 key(상태코드)와 doc_count(건수) 확인

# 예상 결과:
#   200: ~420건
#   401: ~95건 (brute-force)
#   404: ~30건
#   301: ~25건
#   403: ~15건
#   500: ~10건

# HTTP 메소드 분포
curl -s 'http://elasticsearch:9200/security-web-*/_search?pretty' \
  -H 'Content-Type: application/json' \
  -d '{"size":0,"aggs":{"methods":{"terms":{"field":"http.request.method.keyword"}}}}'
# aggregations.methods.buckets 에서 확인

# User-Agent 상위 5개
curl -s 'http://elasticsearch:9200/security-web-*/_search?pretty' \
  -H 'Content-Type: application/json' \
  -d '{"size":0,"aggs":{"agents":{"terms":{"field":"user_agent.original.keyword","size":5}}}}'
# aggregations.agents.buckets 에서 확인
# 주목: sqlmap User-Agent가 보이면 공격 도구 사용 흔적!
```

## Step 4: GeoIP 필드 확인

### 4-1. GeoIP 변환 원리

Logstash의 `geoip` 플러그인은 IP 주소를 지리적 위치 정보로 변환한다.

```ruby
geoip {
  source => "source.ip"
  target => "source.geo"
}
```

### 4-2. GeoIP 필드 확인

Kibana Discover에서 다음 필드를 확인한다:

```
source.geo.country_name
source.geo.city_name
source.geo.location.lat
source.geo.location.lon
source.geo.continent_name
```

### 4-3. KQL로 특정 국가 필터링

Discover 검색창에 다음 KQL 쿼리를 입력한다:

```
source.geo.country_name: "China"
```

```
source.geo.country_name: "Russia"
```

> **참고**: 사설 IP(10.x, 172.16.x, 192.168.x)는 GeoIP 변환이 되지 않는다. 샘플 로그에 포함된 공인 IP만 위치 정보가 매핑된다.

### 4-4. 필드 타입 검증

각 필드가 올바른 타입으로 매핑되었는지 확인합니다:

```bash
# tools 컨테이너 내부에서 실행 (docker exec -it tools bash)
curl -s 'http://elasticsearch:9200/security-web-*/_mapping?pretty'
# mappings.properties 에서 각 필드의 type 확인
# 또는 jq로 핵심 필드만 추출:
curl -s 'http://elasticsearch:9200/security-web-*/_mapping' | jq '.[].mappings.properties | {source, http, url}'
```

> **`.keyword` 접미사 설명**: Elasticsearch의 `text` 필드는 분석기(analyzer)를 거쳐 토큰화됩니다.
> Terms aggregation, 정렬(sort), 정확한 값 매칭에는 `.keyword` 서브필드를 사용해야 합니다.
> 예: `source.ip.keyword`, `http.request.method.keyword`, `user_agent.original.keyword`

**기대 타입:**

| 필드 | 타입 | 이유 |
|------|------|------|
| `source.ip` | `text` + `keyword` | 검색 + 집계 모두 필요 |
| `http.response.status_code` | `long` | 숫자 범위 검색 (>= 400) |
| `url.path` | `text` + `keyword` | 전문 검색 + 정확 매칭 |
| `@timestamp` | `date` | 시간 범위 필터, 히스토그램 |
| `source.geo.location` | `geo_point` | 지도 시각화 |

## Step 5: Sysmon ECS 매핑 확인

### 5-1. Sysmon 원본 필드와 ECS 매핑

`logstash/pipeline/sysmon.conf`의 mutate rename 구문을 확인한다.

```ruby
mutate {
  rename => {
    "SourceIp" => "source.ip"
    "DestinationIp" => "destination.ip"
    "DestinationPort" => "destination.port"
    "Image" => "process.executable"
    "CommandLine" => "process.command_line"
    "ParentImage" => "process.parent.executable"
    "User" => "user.name"
    "Hashes" => "file.hash.sha256"
  }
}
```

### 5-2. 매핑 테이블

| Sysmon 원본 필드 | ECS 필드 | 설명 |
|------------------|----------|------|
| `SourceIp` | `source.ip` | 출발지 IP |
| `DestinationIp` | `destination.ip` | 목적지 IP |
| `DestinationPort` | `destination.port` | 목적지 포트 |
| `Image` | `process.executable` | 실행된 프로세스 경로 |
| `CommandLine` | `process.command_line` | 실행 명령줄 |
| `ParentImage` | `process.parent.executable` | 부모 프로세스 경로 |
| `User` | `user.name` | 실행 사용자 |
| `Hashes` | `file.hash.sha256` | 파일 해시값 |

### 5-3. Kibana에서 확인

1. Data View: `security-sysmon` 선택
2. 다음 필드를 컬럼으로 추가:
   - `process.executable`
   - `process.command_line`
   - `process.parent.executable`
   - `user.name`
